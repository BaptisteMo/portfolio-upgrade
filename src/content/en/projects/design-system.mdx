## Executive summary

When I joined Klépierre, the IT department had never worked with a Product Designer in a lasting way. Internal tools had been built over time based on immediate needs: fast to ship, but **heterogeneous** (visuals, components, patterns, conventions). The result was a fragmented experience for internal users, and unnecessary overhead for teams (re-design, inconsistencies, recurring UI debates).

My mission was to **lay the foundations of a cross-product design system**—with a strong constraint: it had to be **adoptable** in a multi-tech ecosystem (Power BI, Blazor, React…), and deliver value quickly to developers.

In under a year, we produced a **UI Kit** and **documentation**, established a design↔dev collaboration model, and set the first milestones toward a more consistent and sustainable ecosystem for internal applications.

## The strategy I put in place

### 1) [Treat the design system as a product for developers](#a-product-for-developers)
### 2) [Deliver fast without starting from scratch](#deliver-fast-without-starting-from-scratch)
### 3) [Drive adoption: standards, docs, and onboarding](#drive-adoption)

## A product for developers

Before designing components, we needed to understand **what developers were actually experiencing** day to day.

I started by exploring the application ecosystem:
- mapping existing tools and tech stacks (Power BI, Blazor, React…)
- understanding usage contexts and constraints per team
- collecting recurring pain points (inconsistencies, arbitrary UI decisions, lack of reference points)

The observation was simple: total freedom made it possible to ship quickly… but it also created a **fragmented experience** and a hidden cost (repeated decisions, reinvented components, unnecessary variations).

The goal wasn’t to “lock” design down, but to **provide a framework**:
- components people can rely on,
- reusable patterns,
- and a shared language to reduce unproductive debates.

---

## Deliver fast without starting from scratch

With limited bandwidth at the beginning, I optimized for **time-to-value**.

### Choice #1 — A tangible pilot ground

Alongside redesign work on *Klub!* (an IT product), I used this context as a pilot to kick-start the design system:
- components that were truly needed,
- real constraints,
- short iteration loops,
- and an initial set solid enough to put in developers’ hands.

### Choice #2 — A proven technical foundation

When a full-stack developer joined the effort (a few hours per week), the main constraint was clear:
- no time to build everything from scratch
- need a healthy, reusable, maintainable architecture

We therefore chose to start from a trusted foundation: **Radix**.  
The goal was to secure accessibility and interaction behaviors, and focus our effort where it matters (styles, themes, patterns, integration).

### Choice #3 — Align design and code

To avoid the “two worlds” problem (Figma on one side, code on the other), I proposed aligning the **Figma variables architecture** with the **dev-side variables architecture**.  
The goal: reduce gaps, enable contribution, and make handoff more reliable.

## Drive adoption

A design system doesn’t exist if no one uses it.

So I designed the system to be:
- **neutral** (adoptable across multiple products and contexts),
- **flexible** (variants and properties designed to avoid forks),
- **theme-ready** (able to match different styles without rebuilding components).

### Design principles

To guide component production, I set three principles:
- **Simple**: easy to integrate in varied contexts
- **Themeable**: able to work across predefined themes
- **Flexible**: avoid unnecessary functional layers and favor clear patterns

### Docs & communication

In parallel with the components, I documented:
- naming logic,
- usage rules,
- variants,
- and integration best practices.

The goal: frictionless onboarding and progressive adoption, project by project.

## Results and tracking

- 🎨 Structured and documented **UI Kit**, ready to use across IT projects
- 🧩 **Design system evolving continuously**, enriched with every real need
- 🤝 Stronger **design↔dev collaboration** (reviews, discussions, integration alignment)
- 💬 **Developer buy-in**: the approach is understood, expected, and perceived as an accelerator
- 🌍 First steps toward broader **UI consistency** across internal products

## Next steps

- Promote the design system more widely within the IT department
- Grow the component base progressively (prioritize recurring patterns)
- Keep refining documentation to support onboarding and contribution
